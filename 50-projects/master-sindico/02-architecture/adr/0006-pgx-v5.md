# ADR-0006 — pgx/v5 como driver PostgreSQL

## Status

accepted — 2026-04-23 (herdado de legado)

## Contexto

PostgreSQL 18 é o DB canônico. Precisamos driver Go performático, com pool, suporte a tipos PG específicos (jsonb, uuid, timestamptz, array, tsvector), prepared statements, tracing OTel.

## Decisão

**jackc/pgx/v5** como driver PG. Pool `pgxpool.Pool` singleton por processo; passado a cada módulo via DI.

- **`database/sql` + `stdlib`** descartado: overhead, perde features PG.
- **OpenTelemetry** via `otelpgx` (otel-contrib).
- **TLS obrigatório** em prod (`sslmode=require` mínimo; `verify-full` M2+).
- **Connection pool size**: `num_cpus * 2 + spare`; ajustar via Prometheus `pool_acquire_duration_p95`.

## Consequências

### Positivas

- Performance top — benchmarks consistentes mostram pgx 2-3× `database/sql`.
- Tipos PG nativos (UUID, JSONB, arrays, tsvector, range) sem conversão manual.
- Prepared statements automáticos.
- Integração com sqlc oficial.
- Suporte LISTEN/NOTIFY (útil para cache invalidation M2+).

### Negativas

- API mais verbosa que `database/sql` (`pgxpool.Conn.Query` vs `db.Query`).
- Debug de pool hangs requer instrumentação (resolvido via otel + Grafana).

## Alternativas consideradas

1. **`database/sql` + `lib/pq`** — simples, mas lento e sem features PG modernas.
2. **`database/sql` + `stdlib` do pgx** — híbrido; perde ganhos pgx nativo.
3. **go-pg** — ORM-like; overhead; comunidade menor.
