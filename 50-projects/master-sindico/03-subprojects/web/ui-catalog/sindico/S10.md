# S10 — Detalhe Ação PD (campos completos + histórico)

## Finalidade
Visualizar todos os campos de uma ação do Plano Diretor + linha do tempo de versões (histórico imutável de edições). Permite editar, ocultar, criar adendo formal e ver publicações vinculadas na LT.

- `vault/90-archive/master-sindico-pdfs-cliente-originais/JORNADA DO SÍNDICO completa.md` (Regras R2/R4)
- `vault/90-archive/from-development-content-2026-04-23/02-Jornadas/Inventario-Telas.md` (S10 PD)

## Persona e ABAC
- **Persona primária**: Síndico (`syndic`)
- **Plan tier mínimo**: `trial`
- **ABAC action**: `master_plan.read` (sempre) · `master_plan.update` · `master_plan.hide` · `master_plan.amend` (com permissão)
- **Restrições**:
  - Edição requer toggle "Criar/editar Plano Diretor".
  - Ocultar: ação não é deletada; flag `hidden=true` (R3).
  - Adendo formal: cria nova entry sem alterar a original (R5 + C6).

## Fluxo da tela
1. Header com título da ação + status badge + responsável.
2. Sidebar/painel direito: dados estruturados (tipo, áreas, natureza, importância, risco categoria/nível, impacto, próxima ação, datas, observações internas).
3. Seção "Atividades vinculadas na Linha do Tempo": lista cronológica de TimelineEntries que referenciam esta ação (drill-down para [[S11]]).
4. Seção "Histórico de versões": lista de mudanças (campo, valor anterior, valor atual, autor, data) — origem auditoria C5.
5. Botões: "Editar" (abre form em modal/drawer pré-preenchido) · "Ocultar" (com confirmação) · "Criar adendo formal" → modal C6 · "Ver na Linha do Tempo".

## Componentes
- `DetailHeader` (título, status, responsável) — link `[[../../design-system]]`
- `KeyValueGrid` (campos estruturados)
- `TimelineRefList` (publicações vinculadas)
- `VersionHistory` (audit trail — append-only)
- `ActionMenu` (Editar, Ocultar, Criar adendo)
- `AmendmentModal` (texto justificativa + descrição voz/texto + anexo opcional)

## Estados
- **Loading**: skeleton.
- **Empty**: histórico de versões vazio = "Sem edições posteriores".
- **Error**: toast + retry.
- **Success**: tudo carregado.

## Integração com backend
| Ação | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Buscar ação | `/api/v1/condominiums/{id}/master-plan/actions/{action_id}` (inferido) | GET | — | objeto completo |
| Versões | `/api/v1/condominiums/{id}/master-plan/actions/{action_id}/history` (inferido) | GET | — | `[{ change, user, date }]` |
| Editar | `/api/v1/condominiums/{id}/master-plan/actions/{action_id}` | PATCH | partial fields | `{ updated_at, version }` |
| Ocultar | `/api/v1/condominiums/{id}/master-plan/actions/{action_id}/hide` | POST | `{ motivo? }` | `{ hidden: true }` |
| Criar adendo | `/api/v1/condominiums/{id}/amendments` (compliance C6) | POST | `{ ref_type: master_plan_action, ref_id, justificativa, descricao, anexos[] }` | `{ amendment_id }` |
| Publicações vinculadas | `/api/v1/condominiums/{id}/timeline?ref_type=master_plan_action&ref_id={id}` | GET | — | `[entries]` |

## Regras de negócio críticas
- **Nada é deletado** (R3). Botão "Ocultar" mantém registro com `hidden=true`.
- **Edição preserva histórico** (R4): cada PATCH gera nova versão; campos antigos arquivados em audit.
- **Adendo formal** é mecanismo único para correção pós-fechamento (R5 + C6).
- Status só muda via TimelineEntry vinculada (Regra R2 PDF) — UI deve refletir estado atual da PD via reator backend, não permitir edit manual de "status".
- Inputs textuais aceitam comando de voz (R2).

## Gaps/ressalvas
- Endpoint `history` separado é inferido — pode estar embutido em `actions/{id}` com `?include=history`.

- Sem decisão se o status pode ser editado manualmente quando ação foi criada antes de ter LT (cenário inicial); assumido status = "Planejada" até primeira publicação vinculada.
