# M10 — Eventos

## Finalidade
Lista de eventos publicados no condomínio (assembleias, reuniões, prestação de contas, comunicados emergenciais, etc.). Permite ao morador dar Ciência ou Confirmar Participação. Notificação ao síndico em ambas as ações.

## Persona e ABAC
- **Persona primária**: Morador
- **Plan tier mínimo**: any
- **ABAC action**: `event.read` + `event.acknowledge` + `event.confirm_participation`
- **Restrições**: só eventos com `visibility=public` aparecem. Confirmar participação tem prazo final (`participation_deadline`) — após, botão fica disabled.

## Fluxo da tela
1. Sistema carrega lista de eventos do condomínio (futuros + recentes).
2. Sistema renderiza cards com:
   - Tipo (13 tipos canônicos: Assembleia Ordinária, Extraordinária, Reunião de Conselho, etc.)
   - Título
   - Período (data início/fim)
   - Local (se houver)
   - Taxa de visualização (counter — para feedback ao morador)
3. Usuário clica num evento → expande/abre detalhe:
   - Descrição
   - Data início, Data fim
   - Local
   - Anexos
   - Botões: Ciente | Confirmo participação
4. Após Ciente: "Registro de ciência realizado." (fica registrado, imutável)
5. Após Confirmo participação: "Participação confirmada. Pode alterar até o prazo final." (notifica síndico)
6. Sistema permite des-confirmar antes do prazo.

### Tipos de evento (PDF síndico TELA 21 — 17+ tipos)
Assembleia Ordinária; Assembleia Extraordinária; Reunião de Conselho; Prestação de Contas; Audiência Jurídica; Enquete Consultiva; Comunicado Operacional; Comunicado Emergencial; Aviso de Manutenção; Apresentação de Projeto; Reunião com Fornecedor; Treinamento; Simulado de Emergência; Interdição Temporária; Mudança de Procedimento; Campanha Institucional; Atualização Normativa; Outro.

- `EventDetailModal` ou expand inline

- Ver [[../../design-system]]

## Estados
- **Loading**: skeleton 5 cards
- **Empty**: "Não há eventos publicados neste condomínio."
- **Error**: retry
- **Success**: cards renderizados; ações com feedback inline
- **Expirado**: evento passou; botões disabled, badge "Encerrado"

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Listar eventos | `/api/v1/institutional/condominiums/{id}/events` | GET | `?visibility=public` | Lista[Event] |
| Ciência | `/api/v1/institutional/events/{id}/acknowledge` | POST | - | Acknowledgment |
| Confirmar participação | `/api/v1/institutional/events/{id}/confirm-participation` | POST | - | Confirmation |
| Cancelar confirmação | `/api/v1/institutional/events/{id}/cancel-participation` | DELETE | - | - |

Endpoints inferidos de `event.go` (13 tipos canônicos enum).

## Regras de negócio críticas
- R5 pós-fechamento: após data_fim do evento, ciência/confirmação ficam imutáveis
- Notificação automática ao síndico em cada ação (ciência ou confirmação) — R9
- "Pode alterar até o prazo final" — explicitar prazo no card
- Eventos privados (visibility=gestão) NÃO aparecem
- Comunicado Emergencial pode disparar push notification (V2)
- Taxa de visualização exposta no S20 (síndico) é métrica de engajamento

## Gaps/ressalvas
- Cliente não definiu prazo de confirmação default (24h antes do evento? Configurável?). Inferir: campo `participation_deadline` no Event, default = data_inicio.
- Cliente não documentou diferença entre "Ciente" e "Confirmo" em termos jurídicos. Tratar Ciente = leu/notificado; Confirmo = comparecerá.
- Comunicados emergenciais devem talvez exigir "Ciente" obrigatório antes de fechar tela (V2).
- Filtros não foram explicitados — ordenar por data ASC com futuros primeiro.
