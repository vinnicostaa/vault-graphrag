# M4 — Identificação da Unidade

## Finalidade
Captura os dados da unidade do morador no condomínio confirmado em M3: tipo (residencial/comercial/vaga garagem), número, bloco/torre, vínculo (proprietário/inquilino), papel (titular/dependente) e foto de identificação tirada no momento. Etapa pré-termo (M5).

## Persona e ABAC
- **Persona primária**: Morador (em onboarding ou novo vínculo)
- **Plan tier mínimo**: any
- **ABAC action**: `membership.create.self`
- **Restrições**:
  - 1 titular por unidade (regra estrutural R3 + R4)
  - Até 2 dependentes por unidade
  - Unidade vazia não permite dependentes (R5)

## Fluxo da tela
1. Sistema renderiza título: "Informe os dados da sua unidade".
2. Usuário preenche campos obrigatórios:
   - Tipo da unidade: Residencial | Comercial (PDF JORNADA acrescenta Vaga de garagem)
   - Número da unidade
   - Bloco/Torre
   - Vínculo: Proprietário (residente / não residente) | Inquilino
   - Papel: Titular | Dependente
   - Foto de identificação (capturada no momento via webcam/câmera)
3. Sistema valida: se já existe titular para a unidade e usuário marcou Titular → bloqueia com mensagem.
4. Se Titular: exibe nota — "Caso exista inquilino ou outro morador, ele deverá ser cadastrado como dependente."
5. Sistema exibe regras visuais (1 titular, até 2 dependentes, unidade vazia bloqueia dependentes).
6. Usuário clica Avançar → vai para M5 (Termo de Ciência).
7. Voltar → retorna a M3.

## Componentes
- `RadioGroupTipoUnidade`
- `InputNumero` + `InputBloco`
- `RadioGroupVinculo`
- `RadioGroupPapel`
- `WebcamCapture` (captura foto live, sem upload de arquivo)
- `RegrasInline` (exibe as 3 regras estruturais)
- `ButtonGroup` (Avançar / Voltar)

## Estados
- **Loading**: durante validação de unidade existente
- **Empty**: estado inicial com campos vazios
- **Error**:
  - Já existe titular: "Esta unidade já possui titular cadastrado. Selecione 'Dependente' ou contate o síndico."
  - Limite de dependentes atingido: "Esta unidade já tem 2 dependentes."
  - Foto não capturada: "Foto de identificação é obrigatória"
- **Success**: todos campos válidos → botão Avançar habilitado

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Verificar disponibilidade unidade | `/api/v1/institutional/units/check` | GET | `?condo_id&number&block` | { has_titular, dependent_count } (inferido) |
| Upload foto + draft membership | `/api/v1/institutional/memberships/draft` | POST | multipart (campos + foto) | DraftMembership |

Endpoints inferidos. Foto pode ser anexada como parte do membership ou identidade separada.

## Regras de negócio críticas
- R3 (PDF Morador): Vínculo com unidade é obrigatório
- R4: Dependentes não podem alterar status, remover titular ou incluir novos dependentes
- R5: Unidade vazia → dependentes removidos automaticamente
- 1 titular por unidade (TELA 4 do ini cliente)
- Até 2 dependentes por unidade
- Foto tirada no momento (não upload de galeria) — anti-fraude
