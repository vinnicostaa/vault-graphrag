# M13 — Meu Vínculo

## Finalidade
Tela read-only com os dados do vínculo atual do morador (Membership): condomínio, unidade, papel (titular/dependente), tipo de vínculo, foto de identificação, data de criação, status. Pivot para ações de gerenciar dependentes (M14) ou encerrar vínculo (M15).

- `vault/90-archive/from-development-content-2026-04-23/contents/####################TELA DO MORADOR#####(1).md` (TELA 2 estendida — gerenciar meus acessos)
- `vault/90-archive/master-sindico-pdfs-cliente-originais/JORNADA DO MORADOR.md` (TELA M1 — botão "Meu vínculo")
- `vault/90-archive/from-development-content-2026-04-23/02-Jornadas/Inventario-Telas.md` (linha 34)

## Persona e ABAC
- **Persona primária**: Morador (titular ou dependente)
- **Plan tier mínimo**: any
- **ABAC action**: `membership.read.self`
- **Restrições**: dependente vê os próprios dados; só titular vê listagem completa de dependentes (em M14).

## Fluxo da tela
1. Sistema carrega Membership ativo do morador no condomínio em contexto.
2. Sistema renderiza dados:
   - Condomínio (nome + ID)
   - Unidade (número, bloco, tipo)
   - Vínculo (Proprietário residente / Proprietário não residente / Inquilino)
   - Papel (Titular / Dependente)
   - Foto de identificação (capturada em M4)
   - Data de criação do vínculo
   - Status (Ativo / Suspenso / Encerrado)
   - Termo aceito em (link para versão arquivada)
3. Botões de ação:
   - Gerenciar dependentes → M14 (só titular)
   - Encerrar vínculo → M15
   - Atualizar foto (V2 opcional)
4. Voltar → retorna a M1 ou M2.

## Estados
- **Loading**: skeleton
- **Empty**: morador sem vínculo (não chega aqui — redireciona M3)
- **Error**: 404 vínculo encerrado/inválido
- **Success**: dados renderizados; botões ABAC-aware (gerenciar dependentes só se titular)
- **Suspenso**: dados visíveis mas botões disabled + alerta

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Carregar meu vínculo | `/api/v1/institutional/memberships/me/{condo_id}` | GET | - | Membership full |
| Recuperar termo | `/api/v1/institutional/lgpd/consents/{id}` | GET | - | Consent record |

Endpoints inferidos de `membership.go` + Compliance C9 (registro LGPD).

## Regras de negócio críticas
- R3 PDF Morador: vínculo é base de acesso
- R4: dependente NÃO pode alterar status, gerenciar dependentes ou remover titular
- Foto e dados imutáveis pelo morador (alteração via novo cadastro/atualização específica controlada)
- Status "Suspenso" pode ocorrer por revogação do síndico ou compliance pendente — exibir motivo
- Trilha de auditoria de tudo (R10 LGPD)

## Gaps/ressalvas
- Cliente não documentou tela de edição de dados pessoais (telefone/foto) — assumir read-only no MVP, atualização via outro fluxo (perfil global de identidade).
- Cliente não definiu se exibe nome do titular para o dependente (e vice-versa). Recomendar: dependente vê nome do titular; titular vê lista (em M14).
- Botão "Atualizar foto" não está no ini — V2 sugerido.
- Distinção entre titular/dependente em UI: badge proeminente no header.
