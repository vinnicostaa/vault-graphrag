# M3 — Buscar Condomínio por ID

## Finalidade
Permite ao morador adicionar acesso a um novo condomínio inserindo o **ID alfanumérico de 8 caracteres** gerado na jornada do síndico (TELA 5 do síndico — Identidade do Condomínio). Inclui passo de confirmação visual antes de prosseguir para identificação da unidade.

## Persona e ABAC
- **Persona primária**: Morador (em onboarding ou adicionando segundo vínculo)
- **Plan tier mínimo**: any
- **ABAC action**: `condominium.lookup_by_id`
- **Restrições**: nenhuma na busca; vínculo só será criado após M5 (Termo de Ciência).

## Fluxo da tela
1. Sistema renderiza título: "Adicionar acesso a condomínio".
2. Usuário digita o ID de 8 caracteres alfanuméricos (campo obrigatório).
3. Sistema valida formato (8 chars alfanuméricos, case-insensitive recomendado).
4. Ao submeter: sistema busca o condomínio.
5. Sistema exibe Nome do condomínio + Endereço completo + CEP.
6. Mensagem: "Confirme se este é o condomínio correto antes de prosseguir."
7. Usuário clica Confirmar → vai para M4 (Identificação da Unidade).
8. Usuário clica Voltar → retorna a M2 ou cancela onboarding.

## Componentes
- `InputCodigo` (8 chars, mascarado, com auto-uppercase)
- `CardConfirmacao` (preview do condomínio encontrado)
- `AlertWarning` (aviso de confirmação visual antes de prosseguir)
- `ButtonGroup` (Confirmar / Voltar)
- Ver [[../../design-system]]

## Estados
- **Loading**: spinner durante lookup do ID
- **Empty**: estado inicial, só campo + CTA Buscar
- **Error**:
  - ID inválido (formato errado): "ID deve ter 8 caracteres alfanuméricos"
  - ID não encontrado: "Condomínio não encontrado. Verifique o ID com o síndico."
  - Já vinculado: "Você já tem vínculo neste condomínio. Veja em Selecionar Condomínio." (link para M2)
- **Success**: card com dados + botão Confirmar habilitado

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Buscar condomínio | `/api/v1/institutional/condominiums/lookup` | GET | `?id={code}` | Condominium (nome, endereço completo, CEP) |
| Validar duplicidade | (lado backend, no lookup) | - | - | flag `already_linked` |

Endpoints inferidos de `institutional/condominium_lookup_use_case` (Req 6 onboarding morador).

## Regras de negócio críticas
- Justificativa do cliente: "Confirmação visual reduz risco de erro e fraude"
- ID é único e estável, gerado em S5 (TELA 5 do síndico) com QR Code
- Não cria vínculo ainda — apenas valida o condomínio. Vínculo de fato ocorre após M5 (termo aceito) + M4 (unidade informada)
- LGPD: a busca não expõe dados de moradores; apenas dados públicos do condomínio

## Gaps/ressalvas
- Onboarding-por-Persona menciona busca também por **CEP** ou nome — mas TELA 3 do ini do cliente é estritamente **por ID**. Tratar campo CEP como caminho alternativo opcional, não documentado nesta tela.
- Cliente não documentou estado quando ID está expirado (e.g., mandato encerrado). Tratar como erro genérico "Condomínio indisponível".
