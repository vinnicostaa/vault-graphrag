# M5 — Termo de Ciência de Vínculo

## Finalidade
Tela formal de aceite jurídico antes de criar o Membership. Apresenta declaração de responsabilidade civil/penal, consentimento LGPD e checkbox obrigatório. Após aceite, vínculo é criado e morador tem acesso ao painel (M1).

## Persona e ABAC
- **Persona primária**: Morador (em criação de vínculo)
- **Plan tier mínimo**: any
- **ABAC action**: `membership.accept_terms`
- **Restrições**: aceite cria registro LGPD imutável (usuário, IP, timestamp, versão do termo).

## Fluxo da tela
1. Sistema renderiza título: "Declaração formal de vínculo".
2. Sistema exibe texto completo do termo (ver "Texto canônico" abaixo).
3. Usuário lê o texto (scroll obrigatório recomendado).
4. Usuário marca checkbox: "Declaro que li e concordo".
5. Usuário clica Confirmar e solicitar acesso → backend cria Membership + grava registro LGPD imutável.
6. Sistema redireciona para M1 (Painel) com o vínculo recém-criado como contexto.
7. Voltar → retorna a M4.

### Texto canônico (transcrito do ini cliente)
> Declaro, sob minha responsabilidade civil e penal, que possuo vínculo legítimo com a unidade informada neste cadastro, na condição declarada. Confirmo que as informações prestadas são verdadeiras, completas e atualizadas.
>
> Estou ciente de que a prestação de informações falsas poderá resultar na suspensão ou cancelamento do meu acesso à plataforma, bloqueio da conta e adoção das medidas legais cabíveis.
>
> Declaro estar ciente de que o acesso às informações do condomínio é condicionado à manutenção do vínculo legítimo com a unidade e que alterações no status da unidade poderão impactar meu acesso.
>
> Autorizo o tratamento dos meus dados pessoais para fins de controle de acesso, rastreabilidade e segurança, nos termos da Lei Geral de Proteção de Dados.

## Estados
- **Loading**: ao submeter — spinner no botão
- **Empty**: estado inicial
- **Error**: 
  - Falha de criação: "Não foi possível registrar seu vínculo. Tente novamente."
  - Conflito (titular existe): redireciona para M4 com erro
- **Success**: toast "Vínculo registrado com sucesso" + redirect M1

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Aceitar termo + criar membership | `/api/v1/institutional/memberships` | POST | { draft_id, term_version, accepted: true } | Membership criado |
| Registrar consentimento LGPD | (atomicamente no mesmo POST) | - | { user, ip, ts, version } | LGPDLog id |

Endpoints inferidos do módulo `institutional/memberships` (Req 6 + Req 33-42 cross-domain compliance).

## Regras de negócio críticas
- **R10 (LGPD)**: registrar `{user, ip, timestamp, term_version, consentimento}` de forma imutável
- Termo é versionado — cada nova versão exige re-aceite (D-058 + Compliance C9)
- Após aceite, alterações no status da unidade (titular sai, vínculo encerra, etc.) podem revogar acesso (Req 17 cascateamento)
- Aceite NÃO é reutilizável para outro condomínio — cada vínculo gera aceite próprio
- Imutabilidade: registro de aceite nunca é apagado, apenas marcado como revogado
