# M6 — Linha do Tempo

## Finalidade
Lista cronológica das atividades da gestão visíveis ao morador. Principal fonte de informação do morador (R1 PDF Morador). Apresenta cards com tipo, título, data, importância, status e empresa envolvida.

## Fonte canônica
- `vault/90-archive/from-development-content-2026-04-23/contents/####################TELA DO MORADOR#####(1).md` (TELA 8 — LINHA DO TEMPO)
- `vault/90-archive/master-sindico-pdfs-cliente-originais/JORNADA DO MORADOR.md` (TELA M6)
- `vault/90-archive/from-development-content-2026-04-23/02-Jornadas/Inventario-Telas.md` (linha 27)

## Persona e ABAC
- **Persona primária**: Morador (titular ou dependente)
- **Plan tier mínimo**: any (leitura é base)
- **ABAC action**: `timeline.read` com filtro `visibility=public`
- **Restrições**: morador NÃO publica (R2). Só vê itens onde `visibility = público_aos_moradores` (S15 do síndico).

## Fluxo da tela
1. Sistema carrega timeline do condomínio em contexto, ordenada por data DESC.
2. Sistema aplica filtro server-side: `visibility = public` (esconde "Gestão" only).
3. Sistema renderiza cards cronológicos com:
   - Tipo da atividade (ex: Manutenção, Obra, Comunicado)
   - Título
   - Data
   - Nível de importância (badge colorido)
   - Status (Planejada, Em andamento, Concluída, etc.)
   - Empresa envolvida (se aplicável)
4. Usuário rola/pagina cards.
5. Usuário clica "Ver detalhes" → vai para M7.
6. Filtros opcionais: por período, tipo, status, área.

### Tipos visíveis ao morador (PDF JORNADA M6)
- Atividade da gestão
- Registro de execução
- Comunicado
- Evento
- Adendo
- Resultado de pergunta aos moradores

## Componentes
- `TimelineEntry` (card cronológico)
- `BadgeImportancia` (Baixa/Moderada/Alta/Crítica/Estratégica)
- `BadgeStatus` (8 statuses canônicos enum)
- `BadgeTipo` (12 tipos enum)
- `FilterBar` (período, tipo, status, área impactada)
- `EmptyState`
- `InfiniteScroll` ou paginação
- Ver [[../../design-system]]

## Estados
- **Loading**: skeleton de 5 cards
- **Empty**: "Ainda não há atividades publicadas neste condomínio."
- **Error**: "Não foi possível carregar a Linha do Tempo" + retry
- **Success**: cards renderizados; possível badge "Novo" em entradas não lidas

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Listar timeline pública | `/api/v1/institutional/condominiums/{id}/timeline` | GET | `?visibility=public&page&limit&filter` | Lista[TimelineEntry] |
| Marcar como lido (opcional) | `/api/v1/institutional/timeline/{id}/read` | POST | - | - (inferido) |

Endpoints baseados em `institutional/timeline_entry` (entidade INSERT-only).

## Regras de negócio críticas
- R1 PDF Morador: Linha do Tempo é a principal fonte de informação
- R3 (R3 cross): nada é deletado — apenas oculto/editado com trilha de versão
- R5: pós-fechamento é imutável; alterações via Adendo Formal aparecem como entry separado
- Apenas conteúdo liberado pelo síndico (visibility=public) é exibido
- Empresa envolvida é exibida com badge "verificada" ou "não verificada"
