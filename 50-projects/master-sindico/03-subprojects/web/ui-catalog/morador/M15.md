# M15 — Encerrar Vínculo

## Finalidade
Tela de encerramento voluntário do vínculo do morador com um condomínio. Captura motivo estruturado (4 opções + "Outros" com texto). Se titular: avisa que dependentes serão removidos automaticamente (R5). Ação imutável após confirmação.

## Fonte canônica
- `vault/90-archive/from-development-content-2026-04-23/contents/####################TELA DO MORADOR#####(1).md` (TELA 2 — botão "Excluir (pergunta do motivo de da exclusão...)")
- `vault/90-archive/from-development-content-2026-04-23/02-Jornadas/Inventario-Telas.md` (linha 36)
- `backend/.kiro/specs/.../requirements/institutional.md` Req 17 (cascateamento de revogação)

## Persona e ABAC
- **Persona primária**: Morador (titular ou dependente)
- **Plan tier mínimo**: any
- **ABAC action**: `membership.revoke.self`
- **Restrições**:
  - Confirmação dupla obrigatória (anti-erro)
  - Se titular: cascata remove dependentes (R5)
  - Após encerrar: morador perde acesso ao condomínio mas histórico de auditoria fica imutável

## Fluxo da tela
1. Sistema renderiza título: "Encerrar vínculo com [Nome do Condomínio]".
2. Sistema exibe contexto: condomínio, unidade, papel.
3. Sistema lista 4 motivos (radio):
   - Estou me mudando
   - Vendi a unidade
   - Não quero mais acesso
   - Outros (campo livre obrigatório se selecionado)
4. Se usuário é titular: alerta proeminente — "Ao encerrar, seus dependentes serão removidos automaticamente da unidade."
5. Checkbox confirmação: "Estou ciente de que esta ação é irreversível."
6. Botão Confirmar encerramento (destrutivo, vermelho) → backend revoga membership.
7. Sistema mostra confirmação + redireciona para M2 (lista de vínculos restantes) ou logout se era único vínculo.

### Motivos canônicos (4 + outros)
1. `moving_out` — Estou me mudando
2. `sold_unit` — Vendi a unidade
3. `no_longer_want` — Não quero mais acesso
4. `other` — Outros (texto livre obrigatório)

Nota: spec backend Req 17 lista também `contract_ended` (inquilino) e `removed_by_syndic` (forced) — esses 2 não vêm pelo fluxo M15 (são automáticos).

## Estados
- **Loading**: ao submeter
- **Empty**: estado inicial
- **Error**: "Não foi possível encerrar o vínculo. Tente novamente."
- **Success**: toast + redirect M2 ou logout
- **Confirmação dupla**: checkbox precisa estar marcado para habilitar botão

Endpoints inferidos de `membership.go` + Req 17 cross-domain (cascade revoke).

## Regras de negócio críticas
- R5 PDF Morador: unidade vazia → dependentes removidos automaticamente
- Trilha imutável (R10 LGPD): registro de motivo + ts + ip + user fica permanente
- Encerramento NÃO apaga histórico — apenas marca status=revoked
- Após encerrar: morador perde acesso a M1/M6/M8/etc. do condomínio; mantém histórico nas próprias avaliações já submetidas
- Trilha aparece em S31 (Auditoria) do síndico
- Se vínculo único do morador: logout pós-encerramento

## Gaps/ressalvas
- Cliente não definiu se dependente pode encerrar próprio vínculo sem passar pelo titular. Inferir: sim, pode (autonomia LGPD), mas avisa o titular por notificação.
- Cliente não documentou período de "graça" para reverter (e.g., 24h). Tratar como definitivo.
- Cliente não definiu se síndico é notificado do encerramento — recomendar sim (R9 notificações).
- Motivos do cliente são 4; backend tem 6 (cross-domain Req 17 inclui `contract_ended` e `removed_by_syndic`). Estes 2 últimos são acionados por outras flows, não por M15.
- "Vendi a unidade" pode disparar fluxo de transferência de titularidade (V2 — não previsto no MVP).
