# M14 — Gerenciar Dependentes

## Finalidade
Tela exclusiva do **titular** para gerenciar até 2 dependentes vinculados à sua unidade. Permite convidar (por email/CPF), remover ou re-enviar convite. Aplica regra dura R4: dependente não pode aparecer aqui em modo edição.

## Persona e ABAC
- **Persona primária**: Morador **Titular** (role: `resident_titular`)
- **Plan tier mínimo**: any
- **ABAC action**: `dependent.manage` — gate ABAC condicionado a `role=titular` da Membership
- **Restrições**: 
  - Dependente NÃO acessa esta tela (R4)
  - Máx. 2 dependentes ativos por unidade
  - Unidade vazia (titular saiu) → dependentes removidos auto (R5)

## Fluxo da tela
1. Sistema valida que usuário é titular do vínculo. Se dependente → 403 com mensagem explicativa.
2. Sistema lista dependentes ativos (até 2) + slots vazios.
3. Cada dependente exibe: nome, foto, status (ativo/convite-pendente), data de criação.
4. Botão "Convidar dependente" (habilitado se < 2):
   - Abre modal com campos: Nome, E-mail, CPF (opcional), papel preset = Dependente
   - Sistema envia convite por email com link de aceite
5. Botão "Remover" em cada dependente:
   - Modal de confirmação ("Tem certeza? O dependente perderá acesso.")
   - Backend revoga membership do dependente, mantém trilha
6. Voltar → retorna a M13.

## Componentes
- `DependentSlotList` (até 2 slots — preenchidos ou vazios)
- `InviteModal` (form de convite)
- `ConfirmationModal` (remover)
- `BadgeStatusConvite` (Aceito / Pendente / Recusado)
- `EmptyState` (sem dependentes)
- Ver [[../../design-system]]

## Estados
- **Loading**: skeleton
- **Empty**: 2 slots vazios + CTA "Convidar dependente"
- **Permission denied**: dependente acessando — mostra "Apenas o titular pode gerenciar dependentes."
- **Error**: falha ao convidar / remover — toast retry
- **Success**: convite enviado / dependente removido — confirma e atualiza lista

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Listar dependentes | `/api/v1/institutional/memberships/{id}/dependents` | GET | - | Lista[Dependent] |
| Convidar | `/api/v1/institutional/memberships/{id}/dependents/invite` | POST | { name, email, cpf? } | Invite |
| Remover | `/api/v1/institutional/memberships/{dependent_id}/revoke` | POST | { reason: 'removed_by_titular' } | - |
| Re-enviar convite | `/api/v1/institutional/invites/{id}/resend` | POST | - | - |

Endpoints inferidos. Convite usa fluxo similar ao S5 (convidar usuário) — ver `institutional/invites`.

## Regras de negócio críticas
- R3 PDF Morador: vínculo dependente atrelado a 1 titular
- R4: dependentes não podem alterar status, remover titular ou incluir novos dependentes
- R5: se titular sai (unidade vira vazia) → dependentes removidos automaticamente (cascade)
- Limite hard: 2 dependentes por unidade
- Trilha imutável: remoção registra `revoked_by`, `reason`, `ts`
- LGPD: convite registra consentimento de compartilhamento de email/CPF do convidado

## Gaps/ressalvas
- Cliente não documentou texto exato do email de convite — usar template padrão "Convite Master Síndico — Dependente em [Condomínio]".
- Convite expira? Não definido — sugerir 7 dias.
- Dependente convidado precisa cadastro novo se não tem conta? Sim, fluxo de aceite leva para registro/login.
- Cliente não explicitou se dependente removido pode ser re-convidado depois — assumir sim.
- "Gerenciar meus acessos" no ini do cliente é ambíguo: pode incluir gerenciar próprios dispositivos. Esta tela cobre só dependentes.
