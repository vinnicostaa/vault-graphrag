# M2 — Selecionar Condomínio

## Finalidade
Lista todos os vínculos ativos do morador (múltiplos condomínios + múltiplas unidades) e permite acessar, gerenciar ou excluir cada vínculo. Ponto único de troca de contexto entre condomínios.

## Persona e ABAC
- **Persona primária**: Morador (titular ou dependente)
- **Plan tier mínimo**: any
- **ABAC action**: `membership.list.self`
- **Restrições**: morador só vê próprios vínculos. Excluir vínculo é gate-keepado por confirmação com motivo (M15).

## Fluxo da tela
1. Sistema carrega lista de Memberships ativas + arquivadas do usuário autenticado.
2. Sistema renderiza título: "Selecione o condomínio que deseja acessar".
3. Cada card exibe: Nome do condomínio, ID, Endereço resumido, Unidade, Tipo (Residencial/Comercial), Vínculo (Proprietário/Inquilino/Morador autorizado), Papel (Titular/Dependente), Bloco.
4. Usuário escolhe Acessar → seleciona contexto → vai para M1 (Painel) com aquele condomínio carregado.
5. Botão Editar → abre M14 (Gerenciar dependentes) se titular, ou M13 (Meu Vínculo).
6. Botão Excluir → abre M15 (Encerrar Vínculo) com motivos.
7. Botão geral Adicionar novo condomínio → vai para M3.

## Componentes
- `CardCondominio` (com 7 campos + 3 botões de ação)
- `BadgeRole` (Titular/Dependente)
- `BadgeUnitType` (Residencial/Comercial)
- `EmptyState` (sem vínculos)
- `AddButton` (CTA primário "Adicionar novo condomínio")
- Ver [[../../design-system]]

## Estados
- **Loading**: skeleton cards (3 placeholders)
- **Empty**: "Você ainda não está vinculado a nenhum condomínio." + CTA "Adicionar novo condomínio" (vai M3)
- **Error**: mensagem + retry
- **Success**: lista renderizada, com aviso "Mantenha seus dados atualizados para garantir acesso legítimo."

## Integração com backend
| Ação UI | Endpoint | Método | Payload | Retorno |
|---|---|---|---|---|
| Listar vínculos | `/api/v1/institutional/memberships/me` | GET | - | Lista[Membership + Condominium + Unit] |
| Selecionar contexto | `/api/v1/institutional/memberships/{id}/select` | POST | - | { context_token } (inferido) |
| Excluir vínculo | (delegado a M15 — `/api/v1/institutional/memberships/{id}/revoke`) | POST | { reason } | - |

Endpoints inferidos do módulo `institutional/memberships`.

## Regras de negócio críticas
- R3 (PDF Morador): vínculo é obrigatório para acesso ao condomínio
- Múltiplas unidades + múltiplos condomínios por usuário são suportados (TELA 2 ini cliente)
- Excluir vínculo exige captura de motivo estruturado (M15): estou me mudando, vendi unidade, não quero mais acesso, outros
- R5 (PDF Morador): se titular sai → unidade vira "vazia" → dependentes removidos automaticamente

## Gaps/ressalvas
- Cliente não definiu paginação ou ordenação — inferir: mais recentes primeiro, ativos antes de arquivados.
- Cliente não definiu se há limite de condomínios por morador — não há limite explícito (diferente do síndico que tem hard limit 15).
- Endpoint `/select` é inferência; pode ser feito client-side via state global se backend usar membership_id no header.
