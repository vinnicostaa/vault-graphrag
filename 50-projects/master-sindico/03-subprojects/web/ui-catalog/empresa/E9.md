# E9 — Registrar Execução

## Finalidade
Empresa registra execução de serviço para um condomínio, com ciclo de estados imutável (`rascunho → enviado → aprovado → publicado → bloqueado`). Após validação do síndico, o registro alimenta a Linha do Tempo do condomínio e a memória técnica oficial.

- `vault/90-archive/master-sindico-pdfs-cliente-originais/JORNADA DA EMPRESA-1.md` (TELA E5)
- `vault/90-archive/from-development-content-2026-04-23/contents/####################TELA DO MORADOR#####(1).md` (bloco "🔷 E9 — EXECUÇÃO (REGISTRO DE SERVIÇO)" + estados)

## Persona e ABAC
- **Persona primária**: Empresa (Administrador OU Operador Técnico)
- **Plan tier mínimo**: `plus`
- **ABAC action**: `governance.execution_record.create`
- **Restrições**: empresa precisa ter assinado [[E8]] para o deal alvo; condomínio precisa estar conectado.

## Fluxo da tela
1. Empresa em E1 → clica card "Registrar execução" (ou em [[E7]] detalhe do negócio → "Registrar execução").
2. Sistema abre formulário com seleção do condomínio + tipo de serviço.
3. Empresa preenche todos os campos obrigatórios (lista abaixo).
4. Empresa anexa fotos/vídeos/orçamentos.
5. Empresa clica [Salvar rascunho] (estado=`rascunho`, autosave 30s) ou [Enviar para validação do síndico] (estado=`enviado`).
6. Após enviar: aparece em S22 (Detalhe do registro) para validação síndico.
7. Síndico pode: aprovar (estado=`aprovado` → publica na Linha do Tempo, estado=`publicado`), solicitar ajuste (volta para empresa), rejeitar.
8. Após `publicado`, empresa não edita mais — estado vira `bloqueado para edição`.

## Componentes
- `CondominioSelector` (filtrado por deals ativos)
- `FormStructured` (campos abaixo)
- `RiskClassificationSelector` (Baixo/Moderado/Elevado/Crítico)
- `NaturezaSelector` (Preventiva / Corretiva / Emergencial / Estratégica)
- `FileUploadGallery` (fotos/vídeos)
- `ButtonGroup` (rascunho / enviar)
- `StateBadge`

## Campos
- Condomínio atendido (dropdown filtrado por deals ativos)
- Tipo de serviço executado
- Descrição da execução (texto + voz, R2)
- Área impactada (múltipla, lista padronizada)
- Natureza da atividade (Preventiva/Corretiva/Emergencial/Estratégica)
- Status do serviço
- Data da execução
- Garantia do serviço (em meses ou data fim)
- Orientações técnicas ao condomínio (texto)
- Materiais utilizados (opcional)
- Fotos ou vídeos (link Mux para vídeo, upload S3 para foto)
- Classificação de risco (Baixo/Moderado/Elevado/Crítico)
- Observações

## Estados (canônicos — fonte ####TELA####(1).ini)
1. **Rascunho** — empresa edita livremente
2. **Enviado para validação** — pendente síndico
3. **Aprovado** — síndico validou
4. **Publicado** — visível na Linha do Tempo
5. **Bloqueado para edição** — pós-publicação
- Alternativos: `solicitado_ajuste`, `rejeitado`

## Estados UI
- **Loading**: skeleton form
- **Draft saved**: badge "Rascunho salvo" + autosave
- **Validation-error**: bordas vermelhas em campos obrigatórios
- **Sent**: redireciona para [[E5]] / status de envios
- **Error**: toast retry, conteúdo preservado

## Integração com backend
| Ação UI | Endpoint | Método | Payload |
|---|---|---|---|
| Criar execução | `/api/v1/governance/executions` | POST | full payload |
| Salvar rascunho | `/api/v1/governance/executions/draft` | PUT | partial |
| Enviar | `/api/v1/governance/executions/:id/submit` | POST | - |
| Upload mídia | `/api/v1/media/upload` | POST | multipart |

Use case `governance.CreateExecutionRecord` + integração com Mux (vídeos) e S3 (fotos).

## Regras de negócio críticas
- R4: registro entra como "Aguardando validação" — só publica após síndico (S22)
- R3: pós-publicado, estado fica imutável — alteração só via adendo
- Vídeos com lock 90 dias após upload (D-066 — confirmar)
- Risco "Crítico" dispara notificação prioritária + obriga validação em < 24h
- Após publicado, registro alimenta avaliação institucional ([[E11]]) e dashboard ([[E10]])
