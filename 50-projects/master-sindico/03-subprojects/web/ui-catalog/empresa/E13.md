# E13 — Usuários da Empresa

## Finalidade
Gestão de usuários da empresa na plataforma. Cada usuário possui acesso individual (não compartilhar senha — rastreabilidade obrigatória). Há 2 perfis: **Administrador** (acesso amplo) e **Operador Técnico** (subset de ações operacionais).

## Fonte canônica
- `vault/90-archive/master-sindico-pdfs-cliente-originais/JORNADA DA EMPRESA-1.md` (TELA E12 — Usuários da Empresa)

## Persona e ABAC
- **Persona primária**: Empresa (apenas Administrador)
- **Plan tier mínimo**: `plus`
- **ABAC action**: `enterprise.users.manage`
- **Restrições**: Operador Técnico só visualiza lista; não convida nem remove.

## Fluxo da tela
1. Empresa Administrador em E1 → clica card "Usuários".
2. Sistema lista usuários da empresa (nome, email, perfil, status, último acesso).
3. Administrador clica [Adicionar usuário] → abre modal com formulário (nome, email, perfil).
4. Sistema envia convite por email com token (Zitadel).
5. Usuário convidado aceita convite → cadastra senha → vira usuário ativo.
6. Administrador pode [Remover usuário] (modal double-confirm — usuário fica inativo, não deletado, R3).
7. Administrador pode editar perfil de usuário existente (Admin ↔ Operador).

## Componentes
- `UserTable` (linhas com perfil + status badge)
- `AddUserModal`
- `RoleSelector` (Administrador / Operador Técnico)
- `RemoveUserDialog` (double-confirm)
- `EditUserModal`
- `ResendInviteButton`

## Tipos de usuário (fonte JORNADA DA EMPRESA-1.md)

### Administrador
Acesso completo: todas as telas E1-E16, gestão usuários, gestão agência, perfil, dashboard, propostas.

### Operador Técnico
Acesso restrito a:
- Registrar execução ([[E9]])
- Criar atividade técnica ([[E6]])
- Enviar comunicados (parcial — ver E7 do inventário canônico)
- Publicar vídeos institucionais (somente se NÃO houver agência delegada — confirmar)

## Estados
- **Loading**: skeleton table
- **Empty**: "Apenas você tem acesso. Adicione um operador técnico para delegar execuções."
- **Error**: toast retry
- **Invite-pending**: badge laranja "Convite pendente" + botão [Reenviar convite]
- **Active**: badge verde
- **Removed**: badge cinza "Removido em DD/MM/AAAA" (R3 — não deletado)

## Integração com backend
| Ação UI | Endpoint | Método | Payload |
|---|---|---|---|
| Listar usuários | `/api/v1/enterprise/users` | GET | User[] |
| Convidar usuário | `/api/v1/enterprise/users/invite` | POST | { name, email, role } |
| Remover usuário | `/api/v1/enterprise/users/:id` | DELETE | - |
| Atualizar role | `/api/v1/enterprise/users/:id/role` | PUT | { role } |
| Reenviar convite | `/api/v1/enterprise/users/:id/resend-invite` | POST | - |

Integra com Zitadel para convite + autenticação.

## Regras de negócio críticas
- **Senha individual** — nunca compartilhada (R obrigatória — fonte PDF)
- **R3 nada deletado**: usuários removidos ficam inativos com badge; auditoria preserva ações passadas
- **Limite de usuários**: pendente de definição comercial (Plus = N? Pro = N?). Atualmente sem limite explícito no PDF
- **Operador técnico não envia proposta nem aceita Connect Me** — segrega responsabilidades comerciais
- **MFA obrigatório** para Administrador (M2+ — fonte security.md)
- **Audit trail**: toda ação de gestão de usuário gera log
