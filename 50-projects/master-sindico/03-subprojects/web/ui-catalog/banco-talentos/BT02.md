# BT02 — Vídeo de Apresentação

## Finalidade
Captura/upload do vídeo-currículo do morador. Limite de duração e **lock 90 dias** anti-fraude/anti-spam.

- DB CHECK: `duration_seconds <= 90` (INV-BT-01)

## Estados
- **Inicial**: 2 botões (gravar / enviar).
- **Recording**: contador regressivo, indicador visual.
- **Preview**: player do gravado/enviado + botões "Re-gravar" / "Enviar".
- **Uploading**: progress bar (presigned para Mux).
- **Processing**: spinner "Aguardando processamento Mux".
- **Lock 90d ativo**: tela mostra vídeo atual + texto "Vídeo travado até DD/MM/YYYY (90d). Re-upload disponível em X dias."
- **Rejeitado (>90s)**: toast erro + botão "Re-gravar".
- **Error**: toast retry.

## Fluxo técnico (canônico)
1. Frontend solicita presigned via `POST /videos?type=resume` (GR-019: backend NÃO faz proxy).
2. Cliente envia bytes direto pro Mux.
3. Webhook Mux `video.asset.ready` entrega `duration_seconds` + `playback_id`.
4. Backend valida `duration_seconds <= 90`. Se > 90, marca `Video.state='rejected'` (`duration_exceeded`).
5. Se válido, cria `curriculum_videos(curriculum_id, video_id, duration_seconds, locked_until=now()+INTERVAL '90 days')`.
6. Evento `content.talent.curriculum-updated` publicado.

## Integração com backend
| Ação | Endpoint | Método |
|---|---|---|
| Solicitar presigned | `/api/v1/videos?type=resume` | POST |
| Webhook Mux | `/api/v1/webhooks/mux` | POST (interno) |
| Status do vídeo | `/api/v1/talents/curriculum/me/video` | GET |

## Regras de negócio críticas
- **INV-BT-01**: `duration_seconds ≤ 90` (DB CHECK).
- **INV-BT-02**: `locked_until = video.created_at + 90 days`. Re-upload retorna 409 enquanto `now() < locked_until`.
- Anti-fraude: lock obriga compromisso com a própria imagem; sem currículo-fantasma.
- Sem DRM/privacidade adicional além de signed URL TTL ≤ 24h (INV-070 análogo).
