# Cloudflare D1

**Categoria**: Serverless SQL database (SQLite) acessível de Workers via binding.

**Intenção**: banco relacional pequeno-médio com replicação global de leitura, sem gerenciar servidor, cobrado por operação.

## Quando usar

1. **App CRUD simples** — usuários, posts, comentários, com queries SQL clássicas.
2. **Read-heavy global** — leitura replicada próxima do usuário, write para primary central.
3. **Per-tenant isolation** — 1 DB por tenant (até milhares de DBs; cada < 10GB).
4. **Queue metadata / job tracking** para produtos com Queues.
5. **Metadata queryable para R2** — nome, uploader, tags de arquivo.

## Quando **NÃO** usar

- **OLTP complexo** (joins multi-tabela pesados, ACID cross-region forte, > 100 GB) → Postgres externo via [[../railway/_moc|Railway]] ou RDS, com [[_moc|Hyperdrive (lazy)]] (lazy) para cache de connections.
- **Analytics / OLAP** → ClickHouse, BigQuery, Athena.
- **> 10 GB por DB** — limit hard atual.
- **Stored procedures / extensions** (PostGIS, full-text complexo) → SQLite base é limitado.
- **Latência de write sub-100ms globalmente** — write vai para primary; ler de replica é rápido, escrever não.

## Padrão canônico

### Setup

```bash
# Criar DB
npx wrangler d1 create my-app
# → anota database_id no wrangler.toml

# Migration
npx wrangler d1 migrations create my-app create_users_table
# cria migrations/0001_create_users_table.sql
```

`wrangler.toml`:
```toml
[[d1_databases]]
binding = "DB"
database_name = "my-app"
database_id = "abc123..."
```

`migrations/0001_create_users_table.sql`:
```sql
CREATE TABLE users (
    id         TEXT PRIMARY KEY,
    email      TEXT NOT NULL UNIQUE,
    created_at INTEGER NOT NULL         -- unix epoch ms
);
CREATE INDEX idx_users_email ON users(email);
```

```bash
# Apply
npx wrangler d1 migrations apply my-app                    # local
npx wrangler d1 migrations apply my-app --remote            # prod
npx wrangler d1 migrations apply my-app --remote --env staging
```

### Acesso via Worker

```ts
interface Env { DB: D1Database }

// SELECT single
const user = await env.DB.prepare('SELECT * FROM users WHERE id = ?')
                         .bind(userId)
                         .first<User>()

// SELECT many
const result = await env.DB.prepare('SELECT * FROM users WHERE email LIKE ?')
                           .bind(`%${query}%`)
                           .all<User>()
// result.results (array), result.meta (duration, rows_read, etc.)

// INSERT
await env.DB.prepare('INSERT INTO users (id, email, created_at) VALUES (?, ?, ?)')
            .bind(id, email, Date.now())
            .run()

// Batch (atômico — tudo ou nada)
await env.DB.batch([
  env.DB.prepare('INSERT INTO users VALUES (?, ?, ?)').bind(id1, email1, ts),
  env.DB.prepare('INSERT INTO audit_log VALUES (?, ?, ?)').bind(id1, 'create', ts),
])
```

**Sempre usar `prepare()` + `bind()`** — SQL injection prevention. Nunca interpolar string em query.

## Read replicas (Sessions API)

Desde 2025, D1 tem **Read Replicas** distribuídas globalmente. Worker escolhe réplica mais próxima para reads; writes continuam no primary.

```ts
// Sessions API (read-after-write consistency por session)
const session = env.DB.withSession('first-unconstrained')
const user = await session.prepare('SELECT ...').first()
```

**Consistency modes**:
- `first-unconstrained`: qualquer replica (mais rápido, pode ser stale ~1-2s).
- `first-primary`: primary (consistency forte; mais latência).

## Limites (consultada 2026-04-24)

| Limite | Valor |
|---|---|
| DB size | 10 GB |
| DBs por conta (Workers Paid) | 50k |
| Statements por request | 100k (`batch()`) |
| Query time por statement | 30s |
| Rows read por query | 1000 (default; pode ajustar até 100k com flag) |
| Wall time por request | 30s |

## Preço (consultada 2026-04-24)

| Dimensão | Free | Paid |
|---|---|---|
| Rows read | 5M/dia | $0.001 / 1M |
| Rows written | 100k/dia | $1.00 / 1M |
| Storage | 5 GB total | $0.75 / GB-mo após 5 GB |

**Importante**: D1 cobra **rows read**, não queries. Query que retorna 10 linhas = 10 rows read, não 1.

## Antipatterns específicos

- **`SELECT *` sem filtro em tabela grande** — explode rows read + custo. `SELECT ... LIMIT N` sempre.
- **Migration destrutiva em prod** (`DROP TABLE`, `DROP COLUMN`) — mesmo que SQLite aceite, D1 não tem rollback automático. Usar padrão expand/contract (adicionar coluna nova → backfill → trocar código → remover coluna antiga).
- **Assumir SQLite de disco** — pragmas `journal_mode=WAL`, `synchronous=NORMAL` não aplicam (gerenciados pelo D1).
- **Full-text search com FTS5** — suportado, mas limita `rows_read`; em escala usar serviço dedicado (Algolia, Meili, Typesense).
- **Usar D1 para contador atômico global** — lock contention; melhor Durable Object ([[durable-objects]]) com `state.storage`.
- **> 1000 rows em SELECT sem paginação** — ver [[../../runtime-antipatterns/op-026-lista-sem-paginacao|OP-026]] — padrão universal vale aqui.

## Backup e restore

- **Export**: `wrangler d1 export my-app --remote --output=backup.sql`
- **Import**: `wrangler d1 execute my-app --remote --file=backup.sql`
- **Time-travel**: D1 faz snapshot a cada minuto (últimas 24h). Restore via dashboard.

## Relações

- **Combina com**: [[workers]] (binding); [[kv]] (dado pequeno hot — config, não relacional); [[r2]] (metadata queryable em D1; blob no R2)
- **Alternativas**:
  - **Postgres real** via [[../railway/_moc|Railway]] ou RDS — quando precisa do que SQLite não tem (full-text robusto, PostGIS, extensões). Workers conecta via [[_moc|Hyperdrive (lazy)]] (lazy) para cache de conexão.
  - **[[kv]]** — quando dado é KV simples sem schema.
- **Not-a-replacement**: Postgres complexo, OLAP, vector search nativo
