# Banco de Dados — PostgreSQL + pgx + sqlc + goose

## Stack

- **PostgreSQL 18** — Docker local em dev, RDS em prod
- **pgx/v5** — driver nativo Go (não `database/sql`)
- **sqlc v1.30** — gera código Go tipado a partir de `.sql`
- **goose v3** — migrations via `embed.FS` (dentro do binário)

## Regras Absolutas

- PKs: **UUIDv7** gerado em Go via `pkg/utils` — `gen_random_uuid()` só como default em migrations para linhas internas
- Nomenclatura: `snake_case` em tudo (tabelas, colunas, índices, constraints, enums)
- Toda tabela tem `created_at`, `updated_at`, `deleted_at` — **exceto** tabelas imutáveis por design
- Tabelas imutáveis (sem `deleted_at`, sem UPDATE): `timeline_entries`, `audit_trail`, `assembly_votes`
- Valores monetários: `INTEGER` (centavos) — nunca `FLOAT`, `DOUBLE`, `NUMERIC` para dinheiro
- Fração ideal: `DECIMAL/NUMERIC(10,6)` — nunca `FLOAT`
- Migrations **nunca destrutivas em produção**: sem `DROP TABLE`, sem `DROP COLUMN`, sem `ALTER COLUMN` que perde dados. Em vez disso, deprecar + migrar dados + drop em migration futura.
- **CHECK constraints obrigatórios** para enums e invariantes simples (valor > 0, email não vazio, status IN (...)). Dev legacy ensinou: "validação só na aplicação" cai em produção.
- Sem ORM — pgx direto + sqlc para queries tipadas.

## Estrutura de Migration (goose)

```sql
-- +goose Up
-- +goose StatementBegin
CREATE TABLE subscriptions (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    plan_id             UUID NOT NULL REFERENCES plans(id) ON DELETE RESTRICT,
    status              subscription_status NOT NULL DEFAULT 'trialing',
    price_cents         INTEGER NOT NULL DEFAULT 0 CHECK (price_cents >= 0),
    currency            TEXT NOT NULL DEFAULT 'BRL' CHECK (char_length(currency) = 3),
    current_period_start  TIMESTAMPTZ NOT NULL,
    current_period_end    TIMESTAMPTZ NOT NULL CHECK (current_period_end > current_period_start),
    trial_ends_at       TIMESTAMPTZ,
    canceled_at         TIMESTAMPTZ,
    cancel_at_period_end BOOLEAN NOT NULL DEFAULT FALSE,
    stripe_customer_id      TEXT,
    stripe_subscription_id  TEXT UNIQUE,
    stripe_price_id         TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMPTZ
);

CREATE INDEX idx_subscriptions_user_id ON subscriptions(user_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_subscriptions_status ON subscriptions(status) WHERE deleted_at IS NULL;
CREATE INDEX idx_subscriptions_current_period_end ON subscriptions(current_period_end) WHERE deleted_at IS NULL;
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP TABLE IF EXISTS subscriptions;
-- +goose StatementEnd
```

## goose Especificidades

- Migrations rodam via `database.Migrate(ctx, pool)` no startup (dev) ou `cmd/migrate` em CI/prod
- `//go:embed migrations/*.sql` + `embed.FS` — migrations ficam dentro do binário
- **Em produção, apenas uma instância** roda migration (init container em ECS, ou job exclusivo). goose usa `goose_db_version` como lock, mas paralelismo cria race condition.
- Antes de deploy: `goose fix` no CI converte timestamps (dev) → sequencial (prod)
- Idempotência: `CREATE TABLE IF NOT EXISTS`, `DROP INDEX IF EXISTS`. Teste: `goose up && goose down && goose up` deve passar.
- Após criar nova migration: `go build ./...` — `embed.FS` valida em compile-time

## sqlc — Queries Tipadas

Arquivos de query: `internal/modules/{module}/infrastructure/database/queries/*.sql`
Código gerado: `internal/modules/{module}/infrastructure/database/generated/` — **não editar manualmente**, tem `-- Code generated by sqlc`

```bash
sqlc generate   # raiz do backend
git diff --exit-code internal/**/generated  # deve ser 0 se generated está em dia
```

### Padrão de Query

```sql
-- name: GetSubscriptionByUserID :one
SELECT *
FROM subscriptions
WHERE user_id = $1
  AND deleted_at IS NULL
LIMIT 1;

-- name: ListActiveSubscriptions :many
SELECT *
FROM subscriptions
WHERE status IN ('trialing', 'active')
  AND deleted_at IS NULL
ORDER BY created_at DESC
LIMIT $1 OFFSET $2;

-- name: UpsertSubscription :one
INSERT INTO subscriptions (
    id, user_id, plan_id, status,
    current_period_start, current_period_end,
    stripe_customer_id, stripe_subscription_id, stripe_price_id
)
VALUES ($1, $2, $3, $4, $5, $6, $7, $8, $9)
ON CONFLICT (stripe_subscription_id)
DO UPDATE SET
    status = EXCLUDED.status,
    current_period_end = EXCLUDED.current_period_end,
    updated_at = NOW()
RETURNING *;
```

**Nunca** `SELECT *` em queries de produção — liste colunas explicitamente. Exceção tolerada: em `ON CONFLICT ... DO UPDATE ... RETURNING *` quando o mapper está completo.

### sqlc.yaml — Overrides de Tipos

```yaml
version: "2"
sql:
  - engine: postgresql
    queries: internal/modules/{module}/infrastructure/database/queries
    schema: internal/shared/providers/database/migrations
    gen:
      go:
        package: "{module}db"
        out: "internal/modules/{module}/infrastructure/database/generated"
        sql_package: pgx/v5
        emit_empty_slices: true
        emit_pointers_for_null_types: true   # *string, *time.Time, etc
        overrides:
          - db_type: "uuid"
            go_type: "string"                # UUID sempre string no Go
          - db_type: "uuid"
            nullable: true
            go_type: { type: "string", pointer: true }
          - db_type: "user_role"
            go_type: "string"                # enums como string, sem pgtype.Null*
          - db_type: "user_role"
            nullable: true
            go_type: { type: "string", pointer: true }
          # ... repetir padrão para cada enum do domínio
```

**Por que `*string` em vez de `pgtype.NullUserRole`?** Enums nullable do sqlc com tipos `pgtype.*` criam boilerplate em mappers e quebram serialização JSON. `*string` é nativo, JSON-friendly, e mais simples de testar.

## pgx — Pool e Transação

### Pool Config (pgxpool)

Defaults do pgx não são adequados para produção. Configure explicitamente:

```go
config, err := pgxpool.ParseConfig(dsn)
if err != nil {
    return nil, fmt.Errorf("parse dsn: %w", err)
}

// Production-ready defaults
config.MaxConns        = int32(runtime.NumCPU()*2 + 4)   // ex: 8 cores = 20 conns
config.MinConns        = 4
config.MaxConnLifetime = 15 * time.Minute
config.MaxConnIdleTime = 5 * time.Minute
config.HealthCheckPeriod = 1 * time.Minute

pool, err := pgxpool.NewWithConfig(ctx, config)
```

### Transação — sempre via UnitOfWork

```go
err := uc.unitOfWork.Run(ctx, func(ctx context.Context) error {
    // repositories dentro do callback compartilham mesma tx
    if err := uc.subscriptionRepository.Save(ctx, subscription); err != nil {
        return fmt.Errorf("Save subscription: %w", err)
    }
    if err := uc.auditRepository.Log(ctx, AuditEntry{...}); err != nil {
        return fmt.Errorf("Log audit: %w", err)
    }
    return nil
})
// commit automático se err == nil; rollback automático se err != nil
```

**Internamente, `UnitOfWork.Run` usa `pgx.BeginTxFunc` — não `Begin` manual**. `BeginTxFunc` garante rollback se callback retorna erro ou panic; `Begin` cru esquece rollback em erro.

### DBFromContext no Repository

```go
func (r *SubscriptionRepository) FindByID(ctx context.Context, id string) (*entities.Subscription, error) {
    // DBFromContext devolve a tx ativa se houver, ou o pool
    q := billingdb.New(database.DBFromContext(ctx, r.pool))

    row, err := q.GetSubscriptionByID(ctx, id)
    if err != nil {
        if errors.Is(err, pgx.ErrNoRows) {
            return nil, apperrors.ErrNotFound.WithMessage("subscription id=" + id)
        }
        return nil, fmt.Errorf("GetSubscriptionByID: %w", err)
    }
    return mapRowToSubscription(row), nil
}
```

### Context Timeout (atenção!)

pgx **não faz rollback automático** em context cancelation durante query em execução — apenas em BEGIN/COMMIT. Use timeout explícito em queries potencialmente lentas:

```go
ctx, cancel := context.WithTimeout(ctx, 10*time.Second)
defer cancel()

rows, err := conn.Query(ctx, heavyQuery)
```

## Índices Obrigatórios

Para toda nova tabela:

```sql
-- Soft delete (se tabela usa deleted_at)
CREATE INDEX idx_{table}_deleted_at ON {table}(deleted_at) WHERE deleted_at IS NULL;

-- Tenant isolation
CREATE INDEX idx_{table}_condominium_id ON {table}(condominium_id) WHERE deleted_at IS NULL;

-- Todas as FKs
CREATE INDEX idx_{table}_{fk_column} ON {table}({fk_column});

-- Colunas em WHERE/ORDER BY frequentes
CREATE INDEX idx_{table}_status ON {table}(status) WHERE deleted_at IS NULL;
CREATE INDEX idx_{table}_created_at ON {table}(created_at DESC) WHERE deleted_at IS NULL;
```

Antes de criar o índice em migration: rode `EXPLAIN (ANALYZE, BUFFERS)` via **Postgres MCP Pro** (`explain_plan` com hypothetical indexes) — confirma que o índice é usado e reduz custo.

## MCP Postgres no Fluxo de Desenvolvimento

Postgres MCP Pro em modo `restricted` (ver `steering/mcp-integration.md`):

- `describe_table` — conferir schema depois de migration
- `explain_plan` — com hypothetical indexes, valida impacto **antes** de criar índice real
- `analyze_workload` — recomendação de índices baseada em `pg_stat_statements` (rodar ao fim do sprint)
- `health_check` — detecta bloat, invalid indexes, long queries

**Nunca** aponte o MCP para RDS de produção — use snapshot clone em staging.

## Convenções de Nomenclatura

| Tipo | Padrão | Exemplo |
|---|---|---|
| Tabela | `snake_case` plural | `timeline_entries`, `subscriptions` |
| Coluna | `snake_case` | `condominium_id`, `created_at` |
| PK | `id` (UUID) | `id UUID PRIMARY KEY` |
| FK | `{tabela_singular}_id` | `user_id`, `plan_id` |
| Índice | `idx_{tabela}_{coluna(s)}` | `idx_users_email`, `idx_subscriptions_user_status` |
| Unique constraint | `uq_{tabela}_{coluna(s)}` | `uq_users_email` |
| Check constraint | `chk_{tabela}_{regra}` | `chk_subscriptions_price_positive` |
| Enum | `snake_case` singular | `subscription_status`, `user_role` |
| Trigger | `trg_{tabela}_{ação}` | `trg_subscriptions_update_timestamp` |
