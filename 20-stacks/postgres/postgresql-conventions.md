# PostgreSQL — convenções

Implementação concreta dos padrões de [[../../10-knowledge/database/database-patterns]] em PostgreSQL + pgx + sqlc + goose.

## Stack

- **PostgreSQL 18** — Docker local em dev, managed (RDS/Railway) em prod
- **pgx/v5** — driver nativo Go (**não** `database/sql`)
- **sqlc v1.30+** — gera código Go tipado a partir de `.sql`
- **goose v3** — migrations via `embed.FS` (dentro do binário)

## Estrutura de migration (goose)

```sql
-- +goose Up
-- +goose StatementBegin
CREATE TABLE subscriptions (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    plan_id             UUID NOT NULL REFERENCES plans(id) ON DELETE RESTRICT,
    status              subscription_status NOT NULL DEFAULT 'trialing',
    price_cents         INTEGER NOT NULL DEFAULT 0 CHECK (price_cents >= 0),
    currency            TEXT NOT NULL DEFAULT 'BRL' CHECK (char_length(currency) = 3),
    current_period_start TIMESTAMPTZ NOT NULL,
    current_period_end   TIMESTAMPTZ NOT NULL CHECK (current_period_end > current_period_start),
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at          TIMESTAMPTZ
);

CREATE INDEX idx_subscriptions_user_id ON subscriptions(user_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_subscriptions_status ON subscriptions(status) WHERE deleted_at IS NULL;
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP TABLE IF EXISTS subscriptions;
-- +goose StatementEnd
```

## goose — especificidades

- Migrations rodam via `database.Migrate(ctx, pool)` no startup (dev) ou binário separado `cmd/migrate` em CI/prod
- `//go:embed migrations/*.sql` + `embed.FS` — migrations dentro do binário (imutáveis após build)
- **Em produção, apenas uma instância roda migration**. goose usa `goose_db_version` como lock, mas paralelismo cria race. Usar init container em ECS, ou job exclusivo.
- Antes de deploy: `goose fix` no CI converte timestamps (dev) → sequencial (prod)
- Idempotência: `CREATE TABLE IF NOT EXISTS`, `DROP INDEX IF EXISTS`. Teste: `goose up && goose down && goose up` deve passar
- Após criar nova migration: `go build ./...` — `embed.FS` valida em compile-time

## sqlc — queries tipadas

Arquivos: `internal/modules/{mod}/infrastructure/database/queries/*.sql`
Gerado: `internal/modules/{mod}/infrastructure/database/generated/` (**não editar** — header `-- Code generated by sqlc`)

```bash
sqlc generate                                    # raiz do backend
git diff --exit-code internal/**/generated       # deve ser 0 se gerado está em dia
```

### Padrão de query

```sql
-- name: GetSubscriptionByUserID :one
SELECT id, user_id, plan_id, status, ...
FROM subscriptions
WHERE user_id = $1
  AND deleted_at IS NULL
LIMIT 1;

-- name: UpsertSubscription :one
INSERT INTO subscriptions (...) VALUES (...)
ON CONFLICT (stripe_subscription_id)
DO UPDATE SET
    status = EXCLUDED.status,
    updated_at = NOW()
RETURNING *;
```

**Nunca** `SELECT *` em produção — listar colunas. Exceção: `RETURNING *` quando mapper está completo.

### sqlc.yaml — overrides canônicos

```yaml
version: "2"
sql:
  - engine: postgresql
    queries: internal/modules/{mod}/infrastructure/database/queries
    schema: internal/shared/providers/database/migrations
    gen:
      go:
        package: "{mod}db"
        out: "internal/modules/{mod}/infrastructure/database/generated"
        sql_package: pgx/v5
        emit_empty_slices: true
        emit_pointers_for_null_types: true
        overrides:
          - db_type: "uuid"
            go_type: "string"
          - db_type: "uuid"
            nullable: true
            go_type: { type: "string", pointer: true }
          - db_type: "user_role"
            go_type: "string"
          - db_type: "user_role"
            nullable: true
            go_type: { type: "string", pointer: true }
```

**Por que `*string` em vez de `pgtype.NullUserRole`?** Enums nullable do sqlc com tipos `pgtype.*` criam boilerplate em mappers e quebram serialização JSON. `*string` é nativo, JSON-friendly, simples de testar.

## pgx — pool config

Defaults não são adequados pra produção:

```go
config, err := pgxpool.ParseConfig(dsn)
if err != nil {
    return nil, fmt.Errorf("parse dsn: %w", err)
}

config.MaxConns        = int32(runtime.NumCPU()*2 + 4)
config.MinConns        = 4
config.MaxConnLifetime = 15 * time.Minute
config.MaxConnIdleTime = 5 * time.Minute
config.HealthCheckPeriod = 1 * time.Minute

pool, err := pgxpool.NewWithConfig(ctx, config)
```

## Transação — sempre via UnitOfWork

```go
err := uc.unitOfWork.Run(ctx, func(ctx context.Context) error {
    if err := uc.subscriptionRepository.Save(ctx, subscription); err != nil {
        return fmt.Errorf("Save subscription: %w", err)
    }
    if err := uc.auditRepository.Log(ctx, AuditEntry{...}); err != nil {
        return fmt.Errorf("Log audit: %w", err)
    }
    return nil
})
```

**Internamente**, `UnitOfWork.Run` usa `pgx.BeginTxFunc` — **não** `Begin` manual. `BeginTxFunc` garante rollback se callback retorna erro ou panic; `Begin` cru esquece.

Ver [[../../10-knowledge/patterns/transaction-patterns]] pro padrão completo UoW/Saga.

### DBFromContext no repository

```go
func (r *SubscriptionRepository) FindByID(ctx context.Context, id string) (*entities.Subscription, error) {
    q := billingdb.New(database.DBFromContext(ctx, r.pool))

    row, err := q.GetSubscriptionByID(ctx, id)
    if err != nil {
        if errors.Is(err, pgx.ErrNoRows) {
            return nil, apperrors.ErrNotFound.WithMessage("subscription id=" + id)
        }
        return nil, fmt.Errorf("GetSubscriptionByID: %w", err)
    }
    return mapRowToSubscription(row), nil
}
```

### Context timeout (atenção!)

pgx **não faz rollback automático** em context cancelation durante query em execução — apenas em BEGIN/COMMIT. Timeout explícito em queries potencialmente lentas:

```go
ctx, cancel := context.WithTimeout(ctx, 10*time.Second)
defer cancel()

rows, err := conn.Query(ctx, heavyQuery)
```

## Connection string (prod)

```
postgresql://{user}:{password}@{host}:{port}/{database}?sslmode=require&application_name={app-name}
```

- `sslmode=require` **obrigatório** em prod
- `application_name` pra rastreabilidade em `pg_stat_activity`

## MCP Postgres no fluxo de desenvolvimento

Postgres MCP Pro em modo `restricted` (ver [[../../30-agents/mcp-integration]]):

- `describe_table` — conferir schema depois de migration
- `explain_plan` — com hypothetical indexes, valida impacto **antes** de criar índice real
- `analyze_workload` — recomendação baseada em `pg_stat_statements` (ao fim do sprint)
- `health_check` — detecta bloat, invalid indexes, long queries

**Nunca** aponte o MCP para RDS/Railway de produção — use snapshot clone em staging.
